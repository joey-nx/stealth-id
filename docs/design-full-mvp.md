## Problem Statement

서비스마다 KYC를 반복해야 하고, KYC 데이터가 서비스에 직접 전달되어 개인정보 노출 리스크가 있다. 또한 AI Agent가 자율적으로 활동하는 시대에 Agent의 신원을 검증하고 위임 관계를 증명할 방법이 없다. Privacy Pool 서비스를 운영하기 위해, 개인정보 비노출 + 규제 준수를 동시에 달성하는 identity 인프라가 필수적이다.

## Demand Evidence

- Privacy Pool 서비스의 필수 인프라로서 내부 수요 확인됨
- 반복 KYC가 "정말 고통스러운 부분"이라는 직접적 표현
- KYC(사람) + KYA(Agent) 모두 지원 필요 — 내부 스폰서가 KYC+KYA 통합 데모를 최소 요건으로 요구
- 업계 트렌드: eIDAS 2.0 (2026년 말 EU 의무화), Sumsub KYA 프레임워크 런칭, ERC-8004 메인넷 배포

## Status Quo

현재 솔루션 없음. Privacy Pool Registration Tree(KYC-only, depth 16)만 존재하며:
- Agent 식별 불가
- 서비스 간 KYC 재사용 불가
- 감사 레이어 없음

문제를 인지하고 있으나 아직 해결하지 못한 상태. 포터블 credential에 대한 기존 워크어라운드가 없으므로 경쟁 솔루션 대체 비용 제로. (단, 기존 Privacy Pool Registration Tree의 데이터 마이그레이션은 별도 작업 필요.)

## Target User & Narrowest Wedge

**1차 사용자**: Privacy Pool을 사용하는 서비스와 그 사용자
- 서비스 운영자: 반복 KYC 없이 사용자 자격을 검증하고 싶다
- 사용자: 한 번 KYC하고 여러 서비스에서 개인정보 노출 없이 증명하고 싶다
- AI Agent 운영자: Agent가 인간의 위임을 받아 자율적으로 서비스를 이용해야 한다

**Narrowest Wedge**: Privacy Pool 내부 서비스 간 KYC/KYA 재사용 — 외부 확장 전에 내부 인프라로 먼저 증명

## Constraints

1. ZKP circuit은 Noir로 작성 (기존 Privacy Pool과 일관성)
2. KYC provider는 Sumsub (이미 사용 중)
3. 오프체인 감사 레이어는 FATF Rec 17 + GDPR 동시 충족 필요
4. Agent identity는 ERC-8004 호환
5. 기존 Privacy Pool SDK(`@to-nexus/privacy-pool-sdk`)의 Poseidon2, Merkle Tree 프리미티브 재사용

## Premises

1. **Unified Typed Tree** — Human과 Agent를 같은 Merkle Tree에서 관리. `identity_type` 필드로 구분하되, circuit별 selective disclosure로 규제 대응. 추가 사용자 인터랙션 불필요 — circuit 선택으로 처리.
2. **Sumsub + ZKP 이중 구조** — Sumsub ApplicantSharing API로 생태계 내부 공유 + Off-chain Prover에서 commitment 생성 후 ZKP로 외부 증명.
3. **감사 레이어 MVP 포함** — 이것 없이는 FATF Rec 17 기반 법적 재사용 구조 불성립. 나중에 추가가 아니라 처음부터 설계.
4. **ERC-8004 호환** — Agent Tree는 ERC-8004의 agent identity 표준과 호환되도록 설계.

## Cross-Model Perspective

Codex (gpt-5.4) 독립 검토 결과:

**Steelman**: StealthID는 identity 제품이 아니라 **privacy-preserving trust rail**이다. KYC/KYA를 한 번 하고, 재사용 가능한 credential을 발행하고, 모든 relying service가 unlinkable ZK proof로 자격을 검증하게 한다. 핵심은 이중 보장: happy path에서 최소 공개, enforcement path에서 감사 가능한 공개.

**핵심 통찰**: "Privacy Pool service requires this as infrastructure" — 첫 제품은 "모두를 위한 portable identity"가 아니라 **알려진 buyer, workflow, pain point가 있는 좁은 내부 의존성**이다.

**전제 도전**: 별도 트리(원래 전제 1)가 틀렸다고 주장. credential inheritance 패턴에서 트리를 분리하면 cross-tree bridge proof가 필요 → 복잡도만 증가. **이 논거를 수용하여 Unified Typed Tree로 전제 수정.**

**48시간 프로토타입**: Foundry+Anvil, Noir, Go, SQLite. One typed registry tree, Human registration, Agent registration with sponsor link, 2개 Noir proof (eligible_human, eligible_agent_via_sponsor), service-specific nullifier, 감사 stub.

## Approaches Considered

### Approach A: Vertical Slice (Codex 제안과 유사)
- **Summary**: Privacy Pool 통합에 집중한 하나의 end-to-end 데모. Human 1명 + Agent 1개가 2개 서비스에서 ZKP 증명 + 감사 stub.
- **Effort**: S (1-2주)
- **Risk**: Low
- **Pros**: 빠른 검증, 핵심 가치 증명에 집중, 스폰서 데모에 충분
- **Cons**: SDK/감사 레이어가 stub 수준, 프로덕션 전환 시 재작업 필요
- **Reuses**: Privacy Pool SDK의 Poseidon2, Merkle Tree
- **Completeness**: 6/10

### Approach B: Full MVP Architecture (선택됨)
- **Summary**: 전체 아키텍처 설계 후 구현. Sumsub 실제 통합, ERC-8004 완전 구현, 감사 레이어, SDK 전체 포함.
- **Effort**: L (4-6주)
- **Risk**: Med — 범위가 넓어 일정 지연 가능성
- **Pros**: 프로덕션 수준 결과물, 한 번에 올바르게 구축, 외부 확장 준비 완료
- **Cons**: 시간 소요 크고, Sumsub 실제 통합에서 예상치 못한 이슈 가능
- **Reuses**: Privacy Pool SDK, Sumsub ApplicantSharing API
- **Completeness**: 8/10 (Open questions 해결 후 9/10 가능)

### Approach C: Privacy Pool First
- **Summary**: 기존 Registration Tree 확장에 집중. depth 16→32, identity_type 필드 추가, KYA leaf 타입 추가.
- **Effort**: M (2-3주)
- **Risk**: Low
- **Pros**: 기존 코드 최대 재사용, 점진적 확장
- **Cons**: Sumsub 통합과 감사 레이어가 후순위로 밀릴 수 있음
- **Reuses**: Privacy Pool 코드베이스 직접 확장
- **Completeness**: 7/10

## Recommended Approach

**Approach B: Full MVP Architecture** — 스폰서가 KYC+KYA 통합을 최소 요건으로 요구하고, 감사 레이어가 법적 구조의 전제조건이므로, stub 수준이 아닌 프로덕션 수준으로 한 번에 올바르게 구축한다. 4-6주 투자로 프로덕션 전환 시 재작업을 제거한다.

### 구현 순서

**Team assumption**: 2-3명의 시니어 엔지니어. Phase 1-2는 hard commitment, Phase 3-4는 buffer 포함.

**Phase 1 (Week 1-2): Core Identity Layer**

1. Unified Typed Tree 컨트랙트 — Poseidon2 기반, identity_type 필드 포함

   **Leaf 구조 (상세)**:
   ```
   leaf = poseidon2(identity_type, identifier, metadata, nullifier_seed)
   ```
   | 필드 | Human (type=0) | Agent (type=1) |
   |------|---------------|----------------|
   | `identity_type` | `0` | `1` |
   | `identifier` | `hash(sumsub_applicant_id \|\| user_pubkey)` | `hash(erc8004_agent_address \|\| sponsor_commitment)` |
   | `metadata` | `hash(kyc_level \|\| verification_timestamp \|\| expiry)` | `hash(agent_capabilities \|\| delegation_scope \|\| expiry)` |
   | `nullifier_seed` | 사용자가 로컬에서 생성한 랜덤 비밀값 | Agent operator가 생성한 랜덤 비밀값 |

   - identity_type: 0=Human(KYC), 1=Agent(ERC-8004). (ERC-7710 Delegated는 post-MVP로 연기, enum value 2를 예약만 함)
   - depth 20 (최대 ~1M leaves. Privacy Pool 내부 사용 기준 충분. depth 16→20은 proof 크기 20% 증가, 32 대비 40% 절감)

2. Noir circuits:

   **`prove_membership.nr`** — 트리 멤버십 증명 (유형 비공개)
   - Private inputs: leaf 전체 필드, Merkle path, nullifier_seed, revocation_merkle_path
   - Public inputs: tree root, revocation_root, service_nullifier
   - Proves: leaf ∈ tree, nullifier = hash(nullifier_seed || service_id), leaf ∉ revocation_set (non-membership proof)

   **`prove_human.nr`** — 멤버십 + identity_type==0 증명
   - Private inputs: leaf 전체 필드, Merkle path, nullifier_seed, revocation_merkle_path
   - Public inputs: tree root, revocation_root, service_nullifier, identity_type (공개 — 서비스가 human-only 검증을 요구하므로 의도적 공개)
   - Proves: leaf ∈ tree AND identity_type == 0 AND leaf ∉ revocation_set

   **`prove_agent_with_sponsor.nr`** — Agent의 sponsor 인간 연결 증명
   - Private inputs: agent_leaf 전체 필드, agent_merkle_path, sponsor_leaf 전체 필드, sponsor_merkle_path, delegation_signature
   - Public inputs: tree root, service_nullifier, delegation_scope
   - Proves: (1) agent_leaf ∈ tree AND agent.identity_type == 1, (2) sponsor_leaf ∈ tree AND sponsor.identity_type == 0, (3) agent.identifier contains hash of sponsor_commitment, (4) delegation_signature is valid (sponsor가 agent에게 특정 scope를 위임했음을 서명)
   - **주의**: 이 circuit은 2개의 Merkle membership proof + in-circuit 서명 검증을 포함하므로 proving time이 길어질 수 있음. Week 1에서 이 circuit을 먼저 프로토타입하고 proving time 벤치마크 수행 (타겟: 서버 사이드 native proving, 5초 이내).
   - **서명 방식**: EdDSA on BabyJubJub (circuit-friendly, ECDSA 대비 constraint 수 ~10x 절감). 서명 메시지: `eddsa_sign(sponsor_privkey, hash(agent_address || delegation_scope || expiry_timestamp))`.
   - **Fallback (proving time 초과 시)**: recursive verification 대신 **split proof** 접근 — (1) `prove_sponsor_membership.nr`: sponsor가 트리에 있음을 증명 + linking_commitment 출력, (2) `prove_agent_delegated.nr`: agent가 트리에 있고, linking_commitment과 일치하는 sponsor 위임을 갖고 있음을 증명. 두 proof를 on-chain에서 함께 검증. Recursion 불필요.

3. Service-specific nullifier: `nullifier = hash(nullifier_seed || service_id)` — 동일 credential, 서비스마다 다른 nullifier로 연결 불가

4. On-chain Verifier 컨트랙트
   - **배포 타겟: L2 우선** (Base, Arbitrum 등). L1 Ethereum은 gas cost 때문에 비현실적 — ZKP verification은 ~300K gas, 현재 L1 기준 ~$15-30/tx. L2에서는 ~$0.01-0.05/tx.
   - L1에는 tree root만 주기적으로 anchor (optional)

5. **Credential 해지(Revocation)**:
   - Revocation Nullifier Set: 온체인에 해지된 credential의 nullifier를 기록하는 sparse Merkle tree
   - 검증 시 circuit이 "내 nullifier가 revocation set에 없음"을 추가 증명 (non-membership proof)
   - 해지 사유: KYC 상태 변경 (제재, 사기 발각), credential 만료, 사용자 자발적 해지
   - Agent delegation 해지: sponsor가 온체인 revocation tx 전송 → delegation_signature가 revocation set에 추가

6. **Tree insertion access control**: Credential Issuer가 commitment 등록 시 co-sign 필요. 사용자가 직접 tx를 보내되, Credential Issuer의 서명이 없으면 컨트랙트가 reject. 이로써 유효한 attestation 없이는 트리에 삽입 불가. Gas는 사용자 부담 (또는 Privacy Pool이 meta-tx relay 제공).

**Phase 2 (Week 2-3): Sumsub Integration + Identity Components**

Off-chain 시스템을 3개 컴포넌트로 분리 (신뢰 경계 명확화):

1. **Credential Issuer** (서버 사이드, trusted)
   - Sumsub ApplicantSharing API 통합: `POST /resources/applicants/{applicantId}/shareToken`
   - 검증 결과 수신, Sumsub 서명 검증
   - 검증된 credential을 사용자에게 발급 (서명된 attestation)
   - **Sumsub fallback**: Sumsub API가 요구사항을 충족하지 않을 경우, Sumsub에서 KYC 데이터를 직접 export → Credential Issuer가 자체 서명으로 attestation 발급. 이 경우 FATF Rec 17의 "원 검증자 책임" 요건에 대한 법적 검토 필요.

2. **User Prover** (클라이언트 사이드, trustless)
   - 사용자 로컬에서 실행되는 도구 (CLI 또는 SDK)
   - Credential Issuer의 attestation → commitment 생성
   - 온체인 Merkle Tree에 commitment 등록 (사용자가 직접 tx)
   - nullifier_seed 생성 및 로컬 키스토어 관리
   - **Key recovery**: MVP에서는 "키 분실 = credential 재발급" 정책. 사용자가 Sumsub에서 재인증 후 새 commitment 생성, 이전 commitment는 revocation set에 추가. Post-MVP에서 Shamir secret sharing 또는 social recovery 검토.

3. **Agent Registrar** (서버 사이드)
   - ERC-8004 호환 Agent identity 생성
   - Sponsor(Human) 위임 flow:
     1. Sponsor가 delegation message 서명: `sign(agent_address, scope, expiry)`
     2. Agent Registrar가 sponsor의 commitment 존재를 확인
     3. Agent leaf에 `sponsor_commitment` 포함하여 트리에 등록

**Phase 3 (Week 3-4): Off-chain Audit Layer**

1. 감사 데이터 저장소 (encrypted)
   - commitment → 원본 KYC 데이터 매핑
   - **암호화 모델**: AES-256-GCM per-record encryption. 키 구조:
     - Record key = `HKDF(master_key, commitment)` — commitment별 고유 키
     - Master key: HSM 또는 KMS(AWS/GCP) 관리. 운영팀도 단독 복호화 불가 (2-of-3 threshold 접근)
   - **동의 관리**: 사용자 등록 시 감사 동의 서명 기록 (GDPR Article 6(1)(c) — 법적 의무 근거). 동의 철회 시 데이터 삭제 (단, 법적 보존 의무 기간 내에는 보존)

2. 규제기관 요청 처리 flow
   - 감사 요청 수신 (법적 권한 확인: 법원 명령, 규제기관 공식 요청서)
   - 2-of-3 threshold 키 보유자 승인. 키 보유자 역할: (1) Compliance Officer, (2) CTO/Tech Lead, (3) External Legal Counsel. 키 세레모니 절차는 별도 운영 보안 문서에서 정의.
   - 복호화 → 데이터 제출
   - 감사 로그 기록 (요청자, 시간, 대상, 승인자). 로그 자체는 append-only로 변조 불가.

3. TTL 관리 — credential 유효기간. MVP에서는 **한국 규제 기준 단일 정책** (1년 기본, 고위험 6개월). 다중 관할 지원은 post-MVP.

**Phase 4 (Week 4-6): SDK + Integration**
1. 서비스 통합 SDK (TypeScript/Go)
   - `stealthid.verify(proof, servicePolicy)` — 서비스별 검증 policy 기반 자동 검증
   - `stealthid.generateProof(credential, serviceId, disclosurePolicy)` — 사용자 측 proof 생성
2. Privacy Pool 통합
   - 기존 Registration Tree → Unified Typed Tree 마이그레이션
   - Privacy Pool SDK와의 호환 레이어
3. End-to-end 데모
   - Human: Sumsub KYC → commitment → Service A에서 ZKP 증명 → Service B에서 재사용
   - Agent: ERC-8004 등록 → sponsor 연결 → 서비스 이용
   - 감사: 규제 요청 → 데이터 복호화 → 제출

**Timeline cut list** (시간 부족 시 우선 순위별 삭감):
1. (먼저 삭감) Privacy Pool 마이그레이션 → post-MVP로 연기, 새 트리로 fresh start
2. TypeScript SDK → Go SDK만 우선, TS는 후속
3. HSM/KMS 통합 → MVP에서는 환경변수 기반 master key (프로덕션 전 교체 필수)
4. (마지막 삭감) End-to-end 데모 → 최소한 CLI 기반 데모로 대체

**Failure modes**:
- Sumsub API 장애 → Credential Issuer가 재시도 큐로 처리, 사용자에게 지연 알림
- Proof verification 실패 → 서비스가 사용자에게 재생성 요청, 3회 실패 시 새 commitment 등록 안내
- Audit Layer 불가용 → 감사 요청 큐잉, SLA 내 복구. 크리티컬 — 감사 불가능 기간은 규제 위반 리스크

## Open Questions

1. **Sumsub ApplicantSharing API의 실제 제약사항** — sandbox 테스트 후 확인 필요. 특히 sharing token의 TTL과 검증 데이터의 범위.
2. **Nullifier 설계 세부사항** — service_id 잠정 표준: `keccak256(chain_id || verifier_contract_address)`. 온체인 서비스 레지스트리는 post-MVP. MVP에서는 서비스가 자신의 verifier 컨트랙트 주소를 service_id로 사용.
3. **ERC-8004와 Unified Tree의 구체적 호환 방식** — ERC-8004의 agent metadata를 leaf에 어떻게 인코딩할 것인가?
4. **고위험 거래 시 별도 KYC 재수행 정책** — circuit level에서 risk_level 필드를 추가할 것인가, 서비스 level에서 처리할 것인가?
5. **Privacy Pool Registration Tree 마이그레이션 전략** — 기존 depth 16 트리의 데이터를 새 depth 20 트리로 어떻게 이전할 것인가?
6. **다중 관할 규제 차이** — 한국, EU, 미국의 FATF Rec 17 구현 차이를 감사 레이어에서 어떻게 수용할 것인가?

## Success Criteria

1. **KYC 재사용 데모**: 1명의 사용자가 Sumsub KYC 1회 후, 2개 이상의 서비스에서 ZKP로 자격 증명 성공
2. **KYA 데모**: 1개의 Agent가 Human sponsor 위임을 받아 서비스 이용, 위임 관계가 ZKP로 검증됨
3. **비연결성**: 동일 사용자의 서비스 A proof와 서비스 B proof 간 연결 불가능 (nullifier 분석으로 검증)
4. **감사 가능성**: 규제 요청 시 특정 commitment의 원본 KYC 데이터 복호화 및 제출 성공
5. **스폰서 승인**: 위 4가지를 데모하여 내부 스폰서의 "진행해" 승인 획득

## Distribution Plan

- **On-chain contracts**: L2 (Base, Arbitrum 등) 우선 배포. L1은 root anchoring만. Foundry 기반 CI/CD.
- **Off-chain Prover**: Go 바이너리. GitHub Releases로 배포 또는 Privacy Pool 인프라에 통합.
- **SDK**: npm (`@to-nexus/stealth-id-sdk`) + Go module. GitHub Actions CI/CD.
- **Audit Layer**: 서비스로 운영 (API server). Docker container로 배포. Privacy Pool 인프라와 co-located.

## Reviewer Concerns

3회 adversarial review (5/10→7/10→7.5/10) 결과, 다음 이슈가 구현 전 해결 필요:

**HIGH (구현 차단 가능)**:
1. **Revocation tree 구조 미명세** — "sparse Merkle tree"로 명시했으나 구체적 구성(indexed Merkle tree 등)과 Noir에서의 non-membership proof 구현 가능성 확인 필요. Noir 표준 라이브러리에 sparse MT가 없을 수 있음.
2. **Credential 만료 미집행** — leaf metadata에 `expiry`가 있지만 circuit에서 시간 검증 안 함. `min_timestamp` public input 추가하거나, 만료는 오프체인(revocation set에 추가)으로만 처리한다고 명시 필요.
3. **Dual-key 부담** — EdDSA/BabyJubJub (circuit용) + ECDSA/secp256k1 (Ethereum용) 이중 키 관리 필요. 사용자 UX에 미치는 영향과 키 매핑 flow 구체화 필요.
4. **Noir 프리미티브 가용성** — Poseidon2(Poseidon과 다름), EdDSA/BabyJubJub, sparse MT non-membership proof가 Noir에서 사용 가능한지 확인 필요. 미존재 시 직접 구현 필요하며 일정에 반영 필수.

**MEDIUM (아키텍처 변경 없이 해결 가능)**:
5. **Split proof의 linking_commitment 구성 미정의** — sponsor 신원 유출 방지를 위한 구체적 hash 구성 필요.
6. **On-chain verifier 컨트랙트 인터페이스 미정의** — function signatures, 접근 제어(tree root 업데이트 권한, revocation set 추가 권한) 구체화 필요.

**LOW**:
7. **SDK API 타입 시그니처** — 구현 단계에서 정의 가능.
8. **README와 design doc의 leaf 구조 불일치** — README 업데이트 필요.

## What I noticed about how you think

- "사람을 증명할 뿐만 아니라 Agent에 대해서도 식별하는 레이어를 추가하려 한다" — KYC만으로 충분할 수 있는 상황에서 KYA까지 한 번에 가겠다는 것은, 단순한 문제 해결이 아니라 **미래 아키텍처를 지금 잡겠다**는 의지입니다. ERC-8004가 2026년 1월에 메인넷에 올라간 지 두 달밖에 안 된 시점에서 이걸 포함시키는 건 타이밍 감각이 있습니다.
- Unified Tree vs 별도 트리 질문에서 처음에는 별도 트리를 선택했지만, Codex의 cross-tree bridge proof 논거와 selective disclosure 설명을 듣고 **전제를 수정**했습니다. 의견을 바꿀 수 있다는 건 확신과 유연성이 균형을 이루고 있다는 뜻입니다.
- "MVP부터 감사 레이어 포함"에 즉시 동의한 것 — 규제를 나중에 생각할 문제가 아니라 설계의 전제조건으로 본다는 것. 이건 컴플라이언스를 진지하게 다루는 팀의 신호입니다.
- Full MVP Architecture를 선택한 것 — Vertical Slice의 유혹을 뿌리치고 "한 번에 올바르게"를 선택. 스폰서에게 stub이 아닌 실제를 보여주겠다는 판단.
