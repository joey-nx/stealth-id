# PR 생성/업데이트

Pull Request를 생성하거나 기존 PR을 업데이트합니다. 코드 변경사항이 포함된 경우 품질 검사를 먼저 수행합니다.

## 절차

1. **브랜치 상태 확인**
   - 현재 브랜치와 base 브랜치(master) 확인
   - `git log master..HEAD`로 포함될 커밋 목록 확인
   - `git diff master...HEAD`로 전체 변경사항 확인

2. **기존 PR 확인**
   - `gh pr view --json number,title,body,url` 실행
   - PR이 존재하면 업데이트 모드, 없으면 생성 모드로 진행

3. **변경사항 분석**
   - 변경이 코드(`.sol`, `.s.sol`, `.t.sol`)인지 문서(`.md`, `.json`, `.toml`)인지 판별

4. **문서-코드 일관성 검토**
   - 변경된 코드와 관련된 문서(README, 스펙, NatSpec 주석) 확인
   - 코드 변경사항이 문서 내용과 일치하는지 검토
   - 불일치 발견 시 사용자에게 보고하고 수정 필요 여부 확인

5. **코드 변경 시 품질 검사** (문서만 변경된 경우 건너뜀)
   - `forge fmt` - 코드 포맷팅 확인 및 적용
   - `forge build` - 빌드 성공 확인
   - `forge test` - 테스트 통과 확인
   - 검사 실패 시 PR 생성/업데이트 중단하고 문제 보고

6. **커밋되지 않은 변경사항 처리**
   - 스테이징되지 않은 변경사항이 있으면 사용자에게 확인
   - 필요 시 `/commit` 절차 수행

7. **원격 브랜치 푸시**
   - 로컬 커밋이 원격에 없으면 `git push -u origin <branch>` 실행
   - 기존 PR 업데이트 시에도 새 커밋 푸시

8. **PR 생성 또는 업데이트**

   ### 새 PR 생성 (PR이 없는 경우)
   - `gh pr create` 사용
   - 제목: 브랜치의 주요 변경사항 요약
   - 본문 형식:
     ```
     ## Summary
     <1-3 bullet points>

     ## Test plan
     [테스트 체크리스트]

     🤖 Generated with [Claude Code](https://claude.com/claude-code)
     ```

   ### 기존 PR 업데이트 (PR이 있는 경우)
   - 새 커밋 내용을 분석하여 제목/본문 업데이트 필요 여부 판단
   - 변경 범위가 달라졌으면 `gh pr edit --title --body` 사용
   - 본문 업데이트 시 기존 내용에 새 변경사항 반영
   - 단순 버그 수정/개선이면 제목/본문 유지 (푸시만 수행)

9. **결과 확인**
   - PR URL 출력
   - PR 상태 확인 (생성됨/업데이트됨 명시)
